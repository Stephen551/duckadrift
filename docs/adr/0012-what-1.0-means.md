# ADR-0012: What 1.0 means

## Status

Accepted — 2026-07-03.

## Context

`v0.1.0` is a real, tagged release, not a placeholder — but "1.0" as a version
number carries an expectation duckadrift hasn't earned yet: that the tool is
feature-complete against its own governing principle (ADR-0001), not just
useful. Without a written definition, "1.0" could quietly become whichever
release feels ready, which is exactly the kind of undeclared, untested claim
this tool exists to catch in other people's logs. Its own version number
doesn't get an exemption.

Two things get conflated if this isn't written down: shipping more features,
and reaching feature-completeness against the Pact. duckadrift can ship a lot
between now and 1.0 — more deterministic checks, better dialect support,
more onboarding docs — without any of it being what 1.0 actually means. This
ADR draws that line before the confusion has a chance to set in.

## Decision

1.0 ships when the Pact (ADR-0001) is fully live and measured, not merely
implemented. Five conditions, all required, none substitutable:

1. **The semantic check catalog exists and has demonstrated its flagship
   catch** — the checks that read an ADR's substance rather than its
   structure, proven against a real, recurring pattern of decisions that
   keep re-litigating the same unresolved question.
2. **Calibration is real, not typed.** Every confidence threshold that
   decides whether a finding can interrupt anyone is earned from a labeled
   corpus of real findings, hand-verified true or false, and is specific to
   the exact model and inference settings producing it. A number a human
   typed into a config file is not a calibrated threshold — it's a guess
   wearing a threshold's clothes.
3. **A semantic finding interrupts someone for the first time. This is the
   1.0 event itself, not a footnote to it.** Every semantic finding stays
   pull-based (visible in the report, never pushed at anyone) until its
   measured precision, on the labeled corpus, clears the bar its declared
   severity demands. Before that measurement exists, the siren doesn't
   sound on a guess, no matter how confident the guess sounds.
4. **Both ways of paying for inference actually work**, each calibrated
   independently. A repo using an API key and a repo using a Claude
   subscription both get a real, measured guarantee — one is never
   authorized by the other's numbers.
5. **The public contract freezes.** Report format, config keys, ADR
   frontmatter vocabulary, and CLI exit codes stop being negotiable from
   this release forward. After 1.0, changing any of them is a breaking
   change, not a quiet adjustment.

**What 1.0 explicitly does not require:**

- **Adoption.** How many repos use duckadrift is a business question, not
  an engineering one. A tool can be 1.0 with one user or with a thousand;
  the number doesn't move the version.
- **Everything on the roadmap.** Anything past the five conditions above —
  new run modes, other CI platforms, additional integrations — ships as a
  1.x release, never as a precondition of reaching 1.0 in the first place.
- **Authoring or auto-fixing anything, ever.** This isn't a restriction that
  lifts once the tool matures. duckadrift verifies ADR logs; it does not
  write them and it does not silently patch the drift it finds. A future
  version that did either would not be a more complete duckadrift — it
  would be a different tool that happens to share a name.

The 1.0 announcement makes exactly one claim, and it's the same kind of
claim this tool makes about everything else: measured, not asserted. "Measured
precision meets its declared floor, on a labeled corpus" is a falsifiable
sentence, and it's the only sentence 1.0 gets to lead with. A verification
tool doesn't get to launch its own biggest number on faith.

## Consequences

- `v0.1.0` and every release before 1.0 are real, useful, and explicitly not
  1.0 — the deterministic checks this tool ships today already do real work
  without any of the five conditions above being true yet.
- Any pull request or release that changes the report format, a config key,
  the frontmatter vocabulary, or a CLI exit code before 1.0 is not yet
  breaking a promise — the promise starts at 1.0, not before. The same
  change after 1.0 is a breaking change and versioned accordingly.
- A semantic (non-deterministic) finding must never reach anyone through an
  interrupting channel until this ADR's third condition is met, regardless
  of how promising an early prototype looks in testing.
