# ADR-0014: The v0.1.1 post-audit — bucket two shipped unverified

## Status

Accepted — 2026-07-04. Lineage: ADR-0013 (the v0.1.0 adversarial audit).

## Context

ADR-0013 recorded the v0.1.0 audit and committed, in writing, to a specific
safeguard: a full independent verifier re-pass precedes the v0.1.1 tag. Bucket
one — the correctness false positives — got that pass. Bucket two — the
robustness hardening against untrusted fork content — did not. It was
implemented, given its ADR, written into the changelog, and tagged into v0.1.1
on green tests and clean commit messages. The re-pass ADR-0013 promised was run
for one bucket and skipped for the other, and the release shipped anyway.

A tool whose founding claim is that a decision must be checked against reality
tagged a public release on security fixes that were asserted, not checked. That
is the failure ADR-0013 documents, one level up: the process did not run the
check the adversary would run — this time because the check was promised and
then not run, rather than not known.

The missing pass was supplied after the fact, against the public v0.1.1 tag,
from a clean independent clone. It attacked the six shipped robustness fixes
rather than re-running their fixtures. Four held. Two did not.

**Two shipped fixes were incomplete.**

- **Containment was lexical only.** The v0.1.1 fix rejected a link whose path
  text climbed above the repo root, but resolved existence with a call that
  follows symlinks. An in-repo symlink pointing outside the checkout has an
  all-in-repo path and a real target on the runner's filesystem, so a crafted
  link resolved "at HEAD" to a file that is not in the repository at all. The
  plain `../` traversal the fix was built for was caught; the symlink that
  reaches the same place around it was not. The v0.1.1 changelog's claim — that
  a crafted link can no longer reach files outside the repository — was
  therefore not true as shipped.
- **The escaping fix reached one sink of three.** v0.1.1 fenced the reference
  check's link target through the shared helper, but two sibling claims still
  hand-rolled a single-backtick span around an attacker-authorable value: the
  status-graph check's directory label, and the staleness check's review-by
  date. A backtick in a fork-named ADR subdirectory, or in a review-by value
  V8's date parser still accepts as a valid past date, closes the hand-rolled
  span and renders the remainder as live markup in the job summary and the
  scheduled-issue body — the exact injection the fix was meant to close,
  surviving in the sinks the first pass did not reach.

**Four shipped fixes held under direct attack**, and honesty runs both ways:

- The silent-green guarantee held: a genuine mid-walk crash — an over-PATH_MAX
  path, not the broken symlink the fix retired — still wrote a loud failing
  report and exited non-zero, never a silent green.
- The workflow-command surface held: newline and percent escaping reduced an
  injected mask/output-setting payload to inert single-line data.
- The filename-regex fix was linear to 100k characters where the old pattern was
  already at a second by thirty; no other regex in the tree carried the
  catastrophic shape.
- The Action's environment-routing neutralized a shell-metacharacter payload
  that executed when inlined; no attacker-controlled value reaches any run:
  script.

The glob-ReDoS surface ADR-0013 left open as unreproducible was probed again and
remains unreproducible — the library caps brace expansion at a flat cost, not an
exponential one. It stays a documented known-surface, not a fix, for the reason
ADR-0013 gave: this project does not act on a defect no one can demonstrate.

**And the re-pass corrected its own error before it reached the record.** One
check the verifier used to judge the escaping fix — counting backticks and
calling an odd total a breakout — false-flagged a correctly-fenced claim,
because the shared helper legitimately emits an odd count. The heuristic was
discarded and the question re-settled by rendering the claim through a real
CommonMark parser, which showed the payload inert inside a code span. A review
layer that can be wrong and is caught by reproduction is the method, not a hole
in it — the point ADR-0013 made, holding this time for the verifier's own tools.

## Decision

v0.1.1 stays tagged, exactly as shipped — the policy v0.1.0 got, for the reason
v0.1.0 got it. A release that shipped a promise it did not fully keep stays in
history as the record of it. v0.1.2 ships the two completions:

- Containment is re-checked on the canonical path. The resolved target and the
  repo root are both passed through realpath before the containment test, so a
  symlink escape resolves to its real out-of-repo location and is rejected. A
  legitimate in-repo symlink resolves to an in-repo real path and still
  resolves; realpath runs only for links that already exist, and a cycle is
  caught as unresolved rather than a crash.
- Both remaining sinks route through the shared fencing helper — the same one
  the reference check and the directory-label check already use. Output is
  byte-identical for backtick-free values, so no ordinary report changes.

Each fix carries a fixture red on the v0.1.1 tag and green after, verified
independently: the symlink case with a control proving a benign in-repo symlink
still resolves, and a control proving the plain-traversal path stays caught. The
v0.1.2 changelog completes the containment claim rather than silently
re-asserting it — link containment now covers the symlink vector, and it says
so.

The two-bucket vocabulary from ADR-0013 holds: both completions are robustness
against untrusted input, not correctness regressions, and the real-repository
no-regression run confirms it — byte-identical findings against a live external
ADR log, before and after.

## Consequences

- The threat model ADR-0013 expanded is now defended more completely than
  v0.1.1 defended it. The symlink escape and the sibling-sink injection were in
  scope the moment fork content became a supported condition; v0.1.1 covered the
  vector each fix was written against and missed the one beside it. v0.1.2
  covers both.
- The v0.1.1 changelog's containment claim is corrected the only honest way: by
  making it true and saying what changed. A trust tool does not leave a security
  overclaim standing in its own public record.
- An audit run once is not a gate. ADR-0013's re-pass was a promise kept for one
  bucket and skipped for the other, and nothing in the process forced the
  difference to surface before the tag. Whether the pre-tag adversarial
  re-attack becomes a standing, required gate on every public tag — not a step
  that runs when someone remembers to ask — is a director decision, recorded
  here as the finding that motivates it, not ratified here as though the
  verifier could make it. This session is its second data point: a re-attack
  that this time confirmed four fixes and completed two, exactly the work the
  gate would force.
- The audit corpus grows again, permanently: an in-repo symlink escaping the
  repo root, and a backtick in a subdirectory name and in a review-by value. The
  next fix of either class cannot pass a clean exam.
- This post-audit, like the one before it, is a reason to trust the tool. The
  failure it records is not that two fixes were incomplete — incomplete fixes
  are ordinary. It is that a release shipped without the check that would have
  caught them, the check the project had already promised to run. Naming that,
  and building toward the gate that prevents its recurrence, is the founding
  claim applied to the project's own process: checked beats asserted, including
  when the thing left unchecked is the checker's own work.
