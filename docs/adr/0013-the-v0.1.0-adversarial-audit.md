# ADR-0013: The v0.1.0 adversarial audit, and what it exposed

## Status

Accepted — 2026-07-03.

## Context

v0.1.0 shipped after passing every check its own process knew how to run. Then
two adversaries went looking for the checks the process didn't know to run, and
found them. This is the record of that audit, kept because a drift-detection
tool cannot be founded on a quietly-edited account of its own drift.

The audit ran in four passes, each stricter than the last.

**Same-lineage verification passed clean.** Every fix through the pre-launch
calibration work carried a red-before/green-after fixture and a categorized
commit trail. Run against the project's own standard, the tool held.

**Architecture review passed clean.** An independent model (Gemini), given the
codebase to critique, found the structure sound and said so. No structural
rework was indicated.

**The first adversarial pass found crafted-input defects.** A second independent
model (Codex), told to attack rather than review, stopped treating the input as
cooperative. It built inputs the tool was never shown: a link that traverses
above the repository root, a symlink that crashes the walker, a filename that
pins a CPU, markdown that injects live content into the report. Real defects,
all reachable only by an author who is trying to break the tool.

**The second adversarial pass found false positives on ordinary input** — the
serious ones. Not crafted attacks: a capitalized `Accepted` status, a link to a
file whose name contains a space, a changelog line that mentions a retired ADR
number. Each of these makes Tier 0 emit a fact-tier, CI-failing claim that is
simply untrue, on input a normal user would write in their first week. These
break the tool's founding promise — the siren is never wrong — for people who
never did anything unusual.

**And the build process caught its own drift before this could be written up.**
The two adversarial passes ran and were reproduced in the verifier's session,
but were never persisted to the durable record. When the write-up was requested,
the implementer refused to write it from a summary of findings that existed
nowhere it could be checked — and the gap was closed by persisting the raw
reports and reproducing every finding independently, from a clean checkout of
the tagged release, before a word of this ADR was written. The tool's own
discipline, turned on the tool's own process: do not certify a claim you cannot
re-derive from the evidence.

**And independent reproduction corrected the verification layer itself.** One
finding, as first reported, did not reproduce: a malformed-YAML input the
verifier described as passing silently green actually crashed loudly and, with
the silent-green fix in place, produced a loud failing report — not the silent
pass claimed. Reproducing it against the source found the real defect nearby: a
*different* class of frontmatter (absent, or shaped as something other than a
map) is the one silently swallowed, and it was pre-existing, not introduced by
any fix. The reported finding was directionally right and specifically wrong,
and only running it — not trusting it — caught the difference. That a review
layer can be wrong and be corrected by reproduction is not a weakness in the
method; it is the method. A process that depended on flawless reviewers would be
the more fragile one.

Every finding named below was reproduced independently against a clean `v0.1.0`
checkout, with observed output recorded. Nothing here rests on any report's
word alone — including the reports produced by the review passes themselves.

## Decision

v0.1.0 stays tagged, exactly as shipped. The first cut's failure stays in
history on purpose. A real v0.1.1 ships the fixes. The confirmed findings fall
into two buckets, and the split is load-bearing — it is the difference between a
broken promise and an expanded threat model, and the two must not be filed as
one.

**Bucket one — correctness. Tier 0 fired untrue fact-tier claims on ordinary,
non-hostile input. Highest priority.** No attacker is involved in any of these:

- A capitalized `Accepted` — the single most common ADR status, and the
  canonical spelling — was fact-flagged as an invalid status and failed CI.
- A link to a real file whose name contains a space (written percent-encoded,
  the way every renderer expects) was fact-flagged as unresolvable.
- A valid link whose target contains parentheses was truncated mid-path and its
  real target fact-flagged as dangling.
- A changelog line documenting that an old ADR was superseded was read as that
  file "citing the ADR as governing," and failed CI.
- A per-team ADR that supersedes its own directory-local predecessor was
  accused of superseding an unrelated, identically-numbered ADR in a sibling
  directory — because the supersession check resolves numbers globally even
  though this tool's own numbering model, in the very same run, correctly treats
  those numbers as per-directory. The tool contradicted itself between two
  checks.

These are not security findings, and filing them as lower-urgency "hardening"
would be a mistake. They break the never-wrong contract for normal users on day
one. They are fixed first.

**Bucket two — robustness against untrusted input. Honestly, a threat-model
expansion.** duckadrift was designed to run on your own repository, not on
hostile content supplied by a stranger's fork in CI. These findings are real,
and they are outside what the original design defended against, and both of
those things are stated plainly rather than one hidden behind the other:

- A link target is resolved with no repository-root containment, so it can probe
  and "resolve" files anywhere on the runner's filesystem.
- A symlink under the ADR tree crashes the directory walker; because an uncaught
  crash exits without writing a report, the Action then reports a green pass with
  zero findings. The watch stands down, silently and green — the one failure the
  Pact names by name. A malformed frontmatter value and a scalar `governs:` field
  crash the same way, to the same silent-green end.
- A crafted filename triggers catastrophic regex backtracking, pinning a CPU
  until the CI job is killed hours later.
- Crafted markdown injects live autolinks and HTML into the job summary and the
  scheduled-issue body, because the report wraps user-controlled strings without
  escaping them.

Each is fixed, and each is labeled as the robustness hardening it is — the tool
learning to run in a place it wasn't first built to run.

**The deeper finding: the corpus was blind, not just the code.** This is the
point that outlives every individual patch. The pre-launch exam certified "never
cries wolf" against five real open-source ADR logs — a deliberately varied,
genuinely external corpus. It still missed the capitalized status and the
spaced-filename link, because not one of those five logs happened to contain
either. The defects were present the entire time; the exam was too clean to
provoke them. A test corpus assembled from well-kept real repositories is
biased toward well-kept input, and well-kept input is exactly where these bugs
hide. Usefulness-first design built the bugs; a too-friendly corpus then
certified them as absent.

So the permanent remedy is not only the code fixes. The exam corpus itself grows
to include the inputs that ordinary-but-awkward and adversarial reality
produces: mixed-case statuses, spaces and percent-encoding in link targets,
parentheses in paths, history and changelog notes that name dead ADRs, and the
hostile shapes — pathological filenames, symlink cycles, backtick-bearing
content. Every fix in this release adds its own triggering input to that corpus
permanently. The next false positive of this class cannot pass a clean exam,
because the exam is no longer clean.

Mechanics, unchanged from every prior fix: each finding gets its own bisected
commit and an isolating fixture that is red on the tagged v0.1.0 and green after
the fix. Each correctness fix additionally gets a real-repository confirmation,
the same method the pre-launch exam used. A full independent verifier re-pass
precedes the v0.1.1 tag.

## Consequences

- v0.1.0 remains the honest record of what shipped and what it got wrong. It is
  not deleted, moved, or rewritten. The scar is a credential only if it is worn.
- The two-bucket split is permanent vocabulary: a correctness false positive and
  a robustness gap are different severities of different promises, and this
  project will not launder one into the other in either direction.
- The threat model is now explicitly broader than v0.1.0's. Running on untrusted
  fork content is a supported condition as of v0.1.1, not an unconsidered one —
  which means future checks own that condition too.
- The exam corpus is now a first-class artifact that grows on every fix. Its job
  is no longer only to prove a check works, but to stay hostile enough that a
  clean pass means something.
- One flagged surface was deliberately not fixed: a possible glob-ReDoS in the
  library the governed-path check passes patterns to. It was reported as
  unverified, and independent reproduction could not trigger it either — so it
  was not fixed, because the same discipline that governs a finding governs a
  fix. This release does not act on a defect no one can demonstrate. It is
  recorded here, not left silent: the surface is narrowed anyway (the scalar-
  governs normalization means a non-glob value can no longer reach that library
  at all), and it stays a documented known-surface for a future pass that can
  reproduce it. Not fixing what can't be triggered is the method, not a gap in it.
- This audit is the reason to trust the tool, not a reason to doubt it. A
  verifier whose own failures are documented, reproduced, and fixed in public is
  making the exact claim the product makes: checked beats asserted.
